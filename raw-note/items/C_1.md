### 前言

- C 的標準
  - K&R, 1978
  - ANSI C89, 1989
  - ISO C99, 1999
  - ISO C11, 2011
  - _補_, ISO C17, 2017-2018
  - _補_, ISO C23, 2023-2024
- POSIX 標準, POSIX (Portable Operating System Interface)
  - 提供 Unix-like 作業系統共同的基礎
  - 包含 shell, command line tools, C library 等
- POSIX 標準下最常見的實作
  - **BSD**, MacOS, 編譯器: **clang**
  - **GNU**, Linux, 編譯器: **GNU Compiler Collection (gcc)**
- 授權
  - **GNU Public License, GPL**
  - **Lesser GPL, LGPL**
  - **BSD**

### 第一章 - 簡化編譯過程的設定

- 連接第三方函式庫
- 設定基本工具
- 編譯 C 語言
- 設定環境變數, 加入函式庫
- 設定編譯系統

使用套件管理工具

- package repository
- 以作業系統區分
  - Windows: [Cygwin](https://cygwin.com)
  - Mac: [Fink](https://finkproject.org), [Homebrew](https://brew.sh), [Macports](https://macports.org)
- 必須要安裝的套件與工具
  - 編譯器 (compiler), **gcc** 或 **clang**
  - **GBD**, debugger
  - **Valgrind**, 檢查 C 語言記憶體使用錯誤
  - **gprof**, 效能 profiler
  - **make**, 不再需要直接呼叫 compiler
  - **pkg-config**, 尋找函式庫使用
  - **Doxygen**, 生成文件
  - 文字編輯器, **Emacs**, **Vim**, **Kate**, **nano**
  - 或使用 IDE, **Anjuta** (GNOME), **KDevelop** (KDE), **XCode** (Apple), **Code::blocks** (Windows), **Eclipse**
  - Autotools: **Autoconf**, **Automake**, **libtool**
  - **Git**
  - Shell
  - 更多不需要重建輪子的函式庫
    - 例如: `libcURL`, `libGLib`, `libGSL`, `libSQLite3`, `libXML2`

在 Windows 下編譯 C

- C 語言與 Unix 是一起演進的
- 檔案系統主要分成兩大類
  - POSIX 相容系統
  - Windows 家族作業系統
- Windows 下的 POSIX
  - Subsystem for Unix-based Application (SUA)
  - 通常需要使用專業版 Windows 才會提供
- 除此之外需要安裝 **Cygwin**
  - 由 red hat 維護的重編譯相容性程式
- 搭配 POSIX 編譯 C 語言
  - Visual Studio 下原生的 C 語言長期停留在舊版本 (C89)
  - Cygwin 有提供 `gcc`
  - 但是 Cygwin 有 GPL 授權問題 (相依於 `cygwin1.dll` 則必須開源)
- 不搭配 POSIX 編譯 C 語言 (程式不相依於 POSIX 函式)
  - 使用 **MinGW** (Minimalist GNU for Windows)
  - **MSYS** 提供 POSIX shell 可以通過 **mintty** 或 **RXVT** 下使用
  - Cygwin + MinGW gcc
  - Autotools
- _補_: 結論是在 Windows 下受到很多限制, 但是仍然有很多自行建立的空間或解決方案

連結函式庫

- 在 command line 上, 需要幾個步驟
  - 指定編譯器 flag
  - 指定連結的函式庫, 包含在 compiler time 與 link time
  - 在系統上設定以上變數
- 使用 `#include` header
  - 告知編譯器可以檢查簽名
  - 編譯器會檢查函式正確性, 並且產生帶有註記的 object file
  - Linker 的工作就是為了從 object file 中的標記去尋找且連結真正的程式碼
  - `gcc` flags
    - `-l`, 指定連結函式庫
    - `-lc`, 預設會帶有的 flag, 用來連結標準函式庫 **libc**
      - **這個 flag 會依據要連結的函式庫名稱而有所不同**
    - `-o`, 指定編譯後輸出檔的名稱
- 其他常用的編譯器 flags
  - `-g`, 產生除錯用的符號表, 並且並不影響執行速度
  - `-std=gnu11`, 是 clang 與 gcc 特有的 flag, 指定 compiler 接受的 C 版本, 這裡指定是 C11
  - `-O3`, 指定最佳化等級為 3, 使用所有最佳化技術
  - `-O0`, 除錯時可以把最佳化等級降低,
  - `-Wall`, 增加編譯器警告, **務必打開編譯器警告**, 由 compiler 幫忙進行最嚴格的抓錯, 並且要修正所有的錯誤

函式庫路徑

- 函式庫至少有三個可能安裝的路徑
  - 作業系統定義的一到兩個目錄, 用來安裝官方提供的函式庫, POSIX 下的常用路徑
  - 給系統管理員安裝套件的目錄
  - 一般使用者安裝在家目錄下
- 除了系統路徑之外, 其他路徑需要手動去尋找且設定
  - 一旦手動找到後, 應該加入 **makefile** 或者 **shell** 設定上
  - **使用 Shell 環境變數時, 需要注意各個系統的環境變數名稱差異**
  - 使用 **makefile** 更好, 通過 `-I`, `-L` flag 統一設定
- gcc 旗幟
  - `-I`, 指定 header file 的路徑
  - `-L`, 搜尋函式庫的路徑目錄
  - `-l`, 尋找指定函式庫
- 當有多個函式庫並且彼此相依時, **旗幟的順序是重要的, 因為是依序搜尋** (`-l` 的順序)
- 大多數時候, 都可以在常見路徑找到函式庫與標頭檔, 包含作業系統慣例, 套件管理工具慣例
  - `/usr/local`, `/opt`
- POSIX `find` 指令可以協助手動尋找
- 多數時候可以在同目錄下找到 **lib** 以及 **include** 的目錄 (_補_: linux 也是依據此慣例)
- 使用 `pkg-config` 命令, 來搜尋函式庫, 會輸出所需要的包含路徑的 flags
  - `--libs`
  - `--cflags`
  - 可惜的是 `pkg-config` 並不是標準命令, 如果可以使用就很好, 否則仍然需要手動尋找
- 配合 `pkg-config` 的 commend line 小技巧
  - 以 **\`** 包裹 `pkg-config` 指令, 把輸出結果作為取代
- 如果使用環境變數來設定時, 需要**注意**不同的系統 (Linux 或 Mac) 的 `gcc` 會使用不同的變數, 需要小心注意

執行期連結 runtime linker

- static library 由 compiler 負責連結 (link), 並且複製形成最終的執行檔
- shared library 則需要在 runtime 進行連結 (link), 此時會再次面對需要尋找函式庫路徑的問題
- 函式庫位於常見位置
- **Autotools** 打包時, 配合 **Libtool** 加入正確的 flag
- 例如使用 `gcc`, `clang`, 或 `icc` 搭配 libpath 路徑名稱編譯時, 在 makefile 中加入 `LDADD=0Llibpath -W1, -Rlibpath`
- 沒有在常見位置, 也沒有透過 `-W`, `R` flags 指定, 可以通過 shell 的環境變數指定 libpath
  - `export LD_LIBRARY_PATH=libpath:$LD_LIBRARY_PATH`, Linux, Cygwin
  - `export DYLD_LIBRARY_PATH=libpath:$DYLD_LIBRARY_PATH`, OS X

使用 Makefile

- `make` POSIX 指令去執行 `makefile` 中的指令
- 基本上包含變數和一連串的 shell script
- 使用 Makefile 之後就不需要再使用命令的方式直接呼叫編譯器了
- 將檔案名稱為 **makefile** 或 **Makefile** 放置與 `.c` 原始碼同層的目錄中
  - 編譯時只需要呼叫 `make` 即可

設定變數

- 依據 shell 的類型, 設定變數的語法會有所不同
  - 主要分成兩種類型 Bourne shell 和 C shell
  - _補_: C shell 目前已經不再被使用
- shell 使用 `$` 來表達變數, `make` 也使用 `$()` 來表達變數, 但是變數名稱多於一個字元時需要使用 `()`
- 讓 `make` 辨識出變數的方法有以下幾種
  - 在 shell 中使用 `export` 語法輸出變數, 例如: `export CLFAGS='-g -Wall -O3'`
  - 把 `export` 寫在 shell 設定檔裡面, 例如: `.bashrc`, `.zshrc`
  - 把變數指令放在同一句命令之前, 可以只讓變數影響單句命令, 例如: `PANTS=kakhi env | grep PANTS`
  - 把變數寫在 `makefile` 中
  - 使用 `make` 命令上變數, 例如: `make CLFAGS="-g -Wall"`
- C 語言中的環境變數
  - `stdlib` 函式庫裡的 `getenv` 函式
  - POSIX 標準的 `getopt`
  - GNU 標準的 `argp_parse`
- `make` 中的內建變數, POSIX 標準下的內建變數
  - `$@`, target 的完整檔案名稱
  - `$*`, 不含副檔名的 target 名稱
  - `$<`, 觸發建置目標的檔案名稱, 例如: `.c`

makefile 中的規則

- 除了設定變數之外, 都是用以下格式呈現
  - `目標: 相依`
  - ` 命令`, 前面空白是 `tab`
- make 會依據目標與相依性, 決定執行順序
- POSIX 標準下, 將 `.c` 編譯成 `.o`
  - `$(CC) $(CFLAGS) $(LDFLAGS) -o $@ $*.c`
  - `$(CC)` 代表 C 編譯器
- 從目標檔 `.o` 連結 (link) 成可執行檔
  - 對於 linker 來說, 相依的順序很重要
- Flags 是用來提供 compiler 所需要的資訊
- 編譯共享函式庫所需要哪些 linker flags
  - 依據 library 的不同會需要不同的 flags
  - 應該使用工具 `Autotools` 來協助處理
- 使用 `pkg-config` 工具協助尋找對應的 library flags

使用函式庫原始碼

- 編譯由 Autotools 打包的函式庫原始碼
- `wget` 下載原始檔案 `.tar.gz`
- `tar xvzf` 解壓縮 `x` 解壓縮, `v` 詳細版, `z` gzip, `f` 指定檔名
- `cd` 進入解壓縮過後的資料夾中
- `./configure` 檢查相關設定, 尋找是否有缺少的
- `make`
- `sudo make install`, 需要系統管理員權限
- 使用函式庫時需要在 makefile 增加路徑
  - 使用 `pkg-config` 協助尋找
  - 如果不在常見位置或能使用 `pkg-config` 尋找, 就需要手動加上路徑

使用函式庫原始碼 (即使系統管理員不同意)

- 自行在家目錄建立 `root` 目錄
  - `mkdir ~/root`
- 然後在相關路徑上加上新的 root 路徑
  - 例如: 在 shell config 加上 `PATH=~/root/bin:$PATH`
  - 優先搜尋家目錄的 root 以個人管理的版本取代系統標準共用目錄的版本
- 函式庫安裝在自己的家目錄上就不需要系統管理員權限

使用手冊 Manual

- `man -k` 搜尋
- manual 分成多個 sections, 通過 `man man` 可以查詢
  - 1 shell, 例如: `man 1 printf`
  - 3 library, 例如: `man 3 printf`
- 很多文字編輯器或者 IDE 都針對開啟 manual 有快捷鍵
  - 例如: `vim` 中使用 `K` 會顯示當下單字的 manpage

### 第十五章 - 附錄 A: C 語言概述

結構上

- 前置處理器
- 宣告變數和型別
- 函式區塊

編譯

- IDE
- Command line
  - `gcc`, `clang`
  - `make`, `-Wall`, warning all
- 靜態分析, static analysis
  - 想像成是執行前把程式碼交給最好的專家審核

標準函式庫

- 依據路徑, 一連串的清單目錄
- 程式不會單獨存在, 而是與共用的通用函式庫連結

前置處理器

- `.h` 標頭檔 (header file), 提供函式簽名來與二進位函式主體驗證
- 以 `#` 開頭的前置處理器, 主要功能是**字串替換**
  - `#include`, 把名稱替換成指定的函式庫主體

註解

- `/* */` 多行
- `//` 單行

`printf`

- `\n`, 換行
- `\t`, tab
- `%i`, 整數變數
- `%g`, 一般格式
- `%s`, 字串變數
- `%%`, % 符號

變數宣告

- 變數一定要宣告
- 函式也需宣告或定義
  - 宣告與實體可以分離
  - 函式宣告提供資訊給編譯器檢查
  - 因此函式宣告可以分離到 `.h` 標頭檔中
- block `{}`, 形成一個程式碼單元
- 許多 C 程式碼是由下而上的順序
  - 因為使用前必須先定義
  - 因此 `.h` 所有的標頭檔在最上方, 其他函式定義, `main` 函式在最下方
- `void` 表示不回傳任何東西

基本型別聚合成的陣列

- `[]` 陣列
- _index_, 從 `0` 開始的陣列位移值 (offset)
- 以指標 (pointer) 宣告陣列, `*`
- `char *` 表示字串, 也是字元陣列

可以定義新結構型別

- `typedef struct {}`
- 以類似型別轉換的方式, 建立新的結構實體
  - `(){}`, 配合 `.` 指定對應的 numerator 元素

可以知道型別使用多少空間

- `sizeof()`, 回傳指定型別的記憶體容量
- `printf()`, 配合 `%zu` 專屬給 `sizeof` 輸出值使用

字串沒有特別的型別

- 在 C 語言中, 字串就是字元形成的陣列
- 字串結尾的字元 `NULL` 字元, `\0`, 因此字元使用 `''`, 字串使用 `""`
- `strlen()`, 會計算字元數直到 NULL 字元出現但並不包含
- 大多數時候, 字串操作都是通過函式庫進行
  - `string.h`
  - `strlen()`, `strncpy()`, `strncat()`, `strcmp()`, `snprintf()`

表示式 Expression

- 宣告如同名詞, 表示式如同動詞

C 語言生命週期規則

- **scope** 變數範圍
- 全域變數, 定義於函式之外的, 靜態變數 (static)
- **Persistent** 狀態變數,
- 變數宣告在 `{}` scope 內, 在遇到 `}` 後即消除

特殊的 `main` 函式

- 由作業系統呼叫的入口 `main` 函式
- 可以定義成
  - `int main(void)`, `int main()`
  - `int main(int, char**)`, `int main(int argc, char** argv)`
- 回傳值 `0` 代表正常執行, 其他值代表異常
- `main` 會在結尾自動加上 `return 0;`, 可以不需要手動撰寫

C 語言程式主要都是在計算表達式

- 全域變數, 作業系統執行 `main`, 區域變數宣告, 流程控制, 計算表達式

函式使用輸入值副本作計算

- 函式的輸入值都是 copy value

表示式以分號結束

- `;` 代表表示式結束

許多加減乘除變數的方法

- `++`, `--`, `+=`, `-=`, `*=`, `/=`, `%=`

C 語言擴充了 truth 的定義

- **沒有**布林值
- 採用的是 `0`, `'\0'`, `null` 被視為 **false**, 其他皆為 **true**
- `!`, `==`, `!=`, `<`, `<=`, `>`, `>=`, `||`, `&&`
- `||`, 與 `&&` 是延遲計算的
  - 當左側為 **true** 時, 右側會被忽略並不執行計算
  - 範例 `(a < 0 || sqrt(a) < 10)`

兩個整數相除一定會產生整數

- 整數 **integer** 計算速度快, 並且沒有誤差, 因此會盡可能避免使用浮點數 **float point**
- 實數除法 `/`, 分子或分母是實數時採用浮點數除法, 回傳浮點數
- 整數除法 `/`, 分子跟分母都是整數時採用整數除法, 回傳整數
- 取模數 `%`,

三元運算子

- `? :`
- 與 `&&` 和 `||` 一樣都有 short-circuit 延遲計算

分支與迴圈表示式

- 沒有什麼特別之處
- `if`, `else if`, `else`
- `while`, `do while`,
- `for` 只是 while 的精簡版
  - 分成初始化, 測試條件, 步進

指標 pointer

- 也被稱為 alias, reference, label

程式可以直接要求一整塊記憶體

- `malloc` 呼叫配置指定容量的記憶體區塊
  - 需要自行釋放 `free`
  - 在未初始化之前, 內容為未知的垃圾
- 使用指標操作配置好的記憶體區塊

陣列只是記憶體區塊, 所有的記憶體區塊都能視為陣列使用

- 配置的記憶體區塊可以通過 index 操作
- `calloc` 請求記憶體區塊並且同時清除原本的內容

指向常量的指標實際上是一個元素的陣列

- 如果 _i_ 指標指向一個整數, 代表的意義是一個長度為 1 的陣列, _i[0]_ 即可找到 _i_ 指向的位置

指標指向結構內的元素有特殊語法

- 指向 `struct` 的指標內的元素可以通過 `->` 的方式取值
- `pr->numerator` 等同於 `(*pr).numerator`

指標能用來修改函式的輸入參數

- 函式的輸入值都是 copy
- 但是如果輸入值是 pointer 時, pointer 的 copy 仍然指向相同的位址
- 如果 _x_ 是變數時, `&`_x_ 則會取得變數的指標

所有的一切都存在記憶體，指標可以指到這一切

- 函式不能傳入其他函式, 也不能有函式的陣列, 但是可以有**指向函式的指標**
- 通過指向函式的指標來運算或傳遞函式
- `void *`
